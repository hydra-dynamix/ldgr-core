LDGR / ResearchCore

A system for making agents do less, remember better, and prove what changed.

LDGR is a durable coordination layer for AI work. It keeps the model focused on bounded reasoning while the ledger preserves state, evidence, decisions, unresolved issues, and the next concrete action.

Durable Ledger

SQLite-backed project memory, not chat memory.

1. Work itemDefine one bounded piece of work.
2. RunRender context and hand it to an agent.
3. ObserveRecord findings, tests, changes, and risks.
4. AttachPreserve logs, reports, screenshots, outputs, and diffs.
5. DecideClose, continue, split, hold, or queue next work.

The problem LDGR is built around

LLMs are good at local reasoning, synthesis, and implementation. They are weak at maintaining durable state across long tasks. When a task grows, the model starts carrying too much implicit context in its own conversation.

That creates predictable failure modes: forgotten constraints, invented requirements, repeated investigation, false completion, and optimizing for the wrong target.

The model should reason inside the task. It should not be trusted to be the task manager, memory store, judge, and historian at the same time.

Core belief

LDGR splits responsibility. The model can propose, inspect, implement, and explain. The system preserves state, defines boundaries, records evidence, and forces continuation through explicit work items.

This matters because the most dangerous failures are quiet: an agent thinks it completed the assignment when it only completed a convenient substitute, or it makes up constraints that disable itself, or it loses the original reason a decision was made.

1

Bound the work

Each run should complete one coherent work item. Not the whole project. Not a vague pile of progress.

2

Externalize memory

Important facts, observations, artifacts, and decisions live outside model context so they survive restart and handoff.

3

Record evidence

The system preserves what was observed and how it was validated. A claim without evidence is weak.

4

Separate discovery

Agents should not rush into code just because code feels productive. When uncertainty is high, investigation is the work.

5

Leave a next step

A completed run should either close the larger objective or create the next actionable work item.

∞

Repeat safely

`--max-iterations` lets the loop continue, but interventions, failures, unfinished work, and no-pending-work gates stop drift.

What LDGR is deliberately not

Not this

  • A giant prompt that tries to control everything.
  • A benchmark-specific hack that only works once.
  • A fully autonomous agent trusted to decide its own rules.
  • A replacement for human judgment.
  • A system where model-generated summaries become unquestioned truth.

 

But this

  • A small durable structure around agent work.
  • A loop for reducing uncertainty before acting.
  • A provenance layer for what happened and why.
  • A way to keep progress moving across long tasks.
  • A guardrail against drift, false completion, and context collapse.

Typical command flow

ldgr init

ldgr notice add --kind observation \
  --body "Build a durable, testable feature."

ldgr work create first-slice \
  --title "First slice" \
  --description "Small bounded task."

ldgr loop run \
  --prompt prompts/loop-prompt.md \
  --agent-argv '["codex", "exec", "..."]' \
  --max-iterations 10 \
  --stream-agent-output

ldgr decision record first-slice \
  --outcome continue \
  --rationale "Done; next gap found." \
  --next-slug next-slice \
  --next-title "Next slice" \
  --next-description "Follow-up task."

Why evidence can be submitted from anywhere

LDGR-generated loop artifacts go under `.ldgr/artifacts`. When you submit an external report, screenshot, log, dataset, generated file, or output, LDGR records a managed copy under the artifact root so the evidence remains durable and safely served by the cockpit.

The failure LDGR is specifically trying to avoid

The biggest trap is treating model output as authoritative state. A model can summarize incorrectly, invent rules, overweight recent context, or silently mutate the goal. If those outputs are fed back into the system as unquestioned instructions, the system starts enforcing hallucinations.

LDGR’s answer is to make state explicit, inspectable, and tied to evidence. The structure makes it clear what is an observation, what is a decision, what is an artifact, what is an unresolved issue, and what is merely a proposed next step.

LDGR is not trying to make agents magical. It is trying to make their work accountable.

Most organizations do not need magical AI autonomy. They need reliable, inspectable work automation. The value is that an agent can do one bounded thing, prove what happened, and hand off the next thing cleanly.
